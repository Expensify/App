# `expo-video` patches

### [expo-video+55.0.3+001+catch_play_abort_error.patch](expo-video+55.0.3+001+catch_play_abort_error.patch)

- Reason: When rapidly seeking a video via the progress bar on web, `HTMLVideoElement.play()` returns a Promise that gets rejected with `AbortError` if `pause()` is called before it resolves. This patch wraps all 5 `video.play()` call sites in `VideoPlayer.web.js` with `.catch()` to silently swallow `AbortError` while re-throwing any other errors. This is the [standard fix recommended by Chrome](https://developer.chrome.com/blog/play-request-was-interrupted).

### [expo-video+55.0.3+002+fix_settings_popup_position_in_fullscreen.patch](expo-video+55.0.3+002+fix_settings_popup_position_in_fullscreen.patch)

- **Issue**: Android fullscreen video player — opening the Speed/Audio settings popup while the player is in `STATE_ENDED` causes the popover to overlap the playback controls bar. During playback the popup is correctly positioned above the controls.
- **Root cause analysis**: The native ExoPlayer settings popup (`androidx.media3.ui.PlayerControlView.displaySettingsWindow()`) uses `PopupWindow.showAsDropDown(anchorView, xoff, yoff)` anchored to the gear/settings button. The `xoff`/`yoff` offsets (`getWidth() - popupWidth - margin`, `-(popupHeight + margin)`) and the anchor position are calculated fresh each time the gear icon is tapped — the position is NOT cached. However, after `STATE_ENDED` the ExoPlayer controller layout changes internally (play button → replay, controller auto-show mode switches to indefinite). If the controller's `onLayout()` hasn't settled, the anchor view's window position or the controller's measured dimensions used in the offset calculation may be stale, causing the popup to appear overlapping the controls.
- **Changelog** (2026-06-02):
  - **v1 (original)**: Used `playerView.post { playerView.useController = true }` on `STATE_ENDED`.
  - **Verdict on v1**: `PlayerView.setUseController(true)` when the value is already `true` is **a no-op** — the method has `if (this.useController == useController) { return; }` as its first conditional (verified against `androidx/media3/ui/PlayerView.java` source). The patch had no effect on the bug.
  - **v2 (current)**: Calls `playerView.requestLayout()` instead. This forces a fresh measure/layout pass of the entire `PlayerView` when `STATE_ENDED` is reached (via `post()` to ensure it runs after any pending layout). This guarantees that when the user subsequently taps the gear icon, the settings button anchor view and the controller dimensions used by `displaySettingsWindow()` reflect the current controller layout state.
- **Verification**: Patch file path (`FullscreenPlayerActivity.kt`) and insertion point (end of `onPostCreate()`, after the Rectangle hint layout listener) are correct. The import (`androidx.media3.common.Player`) is correct and follows existing import ordering conventions. The logic is sound: `requestLayout()` is the standard Android mechanism to ensure a view hierarchy is laid out with the most recent constraints. Fixes Expensify issue #91216.

# TriNet HR Integration — Full Implementation Plan

## CRITICAL NOTES — Do Not Miss

1. **`updateZenefitsFinalApprover`**: The original design doc did NOT include this function because `GustoFinalApproverPage` hadn't shipped yet when the doc was written. It was added to Gusto later. We MUST include `updateZenefitsFinalApprover` in `Zenefits.ts` (DONE in PR 4) and build `ZenefitsFinalApproverPage` in issue #90583. This mirrors `updateGustoFinalApprover` in `Gusto.ts`.

2. **Issue descriptions may be outdated**: The issue descriptions were written before some Gusto features shipped. Always cross-check against the LATEST Gusto code, not just the issue text — Gusto may have evolved since the issues were created.

3. **Beta removal**: ZENEFITS beta must NOT be removed until ALL releases (1, 2, 3) are shipped and verified on staging. Confirmed by Yuwen after discussion with Brandon and Aldo.

4. **Translations**: Only manually update `en.ts`. All other locale files are auto-generated by the AI translation bot on PR submission. Do NOT add placeholders to non-English files.

5. **`EXPORT_LABELS.ZENEFITS`**: Must stay as `'Zenefits'` (not `'TriNet'`) because `ExportedIconCell` matches against persisted report action labels. `NAME_USER_FRIENDLY.zenefits` is `'TriNet'` for UI display.

6. **`WorkspaceHRPage.tsx` shouldBeBlocked gate**: Do NOT add `|| isBetaEnabled(CONST.BETAS.ZENEFITS)` until the TriNet card UI is actually added to that page (issue #90582). The page is entirely Gusto-wired — a Zenefits-only user would land on a page that can't manage TriNet. Reviewer ShridharGoel caught this in PR #90723 and it was reverted.

---

## Strategic Context

TriNet-connected policies account for $144k in subscription revenue. Porting the integration removes one of the last reasons those customers need Expensify Classic. The project mirrors the existing Gusto HR integration pattern.

## Key Design Decisions

- **Internal name**: `zenefits` everywhere in code (connections, routes, API commands, Onyx keys) — renaming would break existing connectionData, logs, analytics, PHP/Java references
- **User-facing name**: `TriNet` — via `CONST.POLICY.CONNECTIONS.NAME_USER_FRIENDLY.zenefits = 'TriNet'`
- **Backend**: Already functional from Expensify Classic (`ZenefitsSyncJob.java`, OAuth callback at `Web-Expensify/partners/zenefits/`). No backend changes needed for frontend work

## Offline Behavior (All Releases)

- **Config changes** (approval mode, final approver): Optimistic Onyx updates via `pendingFields`/`errorFields`. Changes appear instantly, roll back on failure
- **Connect**: Requires network (OAuth flow). Guarded by `showRequiresInternetModal`
- **Sync**: Requires network. "Sync now" guarded by `showRequiresInternetModal`
- **Disconnect**: `API.write` with optimistic removal of `policy.connections.zenefits`
- **Read-only display** (sync status, approval chains, member counts): Always visible from cached Onyx
